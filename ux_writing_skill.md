# UX 라이팅

제품 전반에서 사용자 중심의 쉽고 명확한 인터페이스 텍스트(마이크로카피)를 설계하고 개선하는 Skill입니다.

버튼, 레이블, 오류/성공 메시지, 알림, 온보딩, 빈 상태, 도움말 등 모든 UI 텍스트를 다루며, 자연스러운 대화 흐름과 일관된 톤과 어조를 함께 설계합니다.

목적성, 간결성, 대화체성, 명확성을 기준으로 작성하고, 접근성과 이해도를 고려해 누구나 쉽게 이해할 수 있는 경험을 만듭니다.

---

## **UX 라이팅**

제품과 사용자 경험을 위해, 쉽고 명확하며 간결한 인터페이스 텍스트(UX 텍스트/마이크로카피)를 작성합니다.
이 과정에서는 사용자가 목표를 달성하는 데 도움이 되는 텍스트를 작성하기 위한 프레임워크, 패턴 및 모범 사례를 제공합니다.

Claude Desktop, Claude Code, Codex(CLI 및 IDE 확장 프로그램) 와 호환됩니다.

---

## **이 스킬은 언제 사용해야 할까요?**

다음과 같은 경우에 이 스킬을 사용하세요:

- 인터페이스 콘텐츠 작성 (버튼, 레이블, 제목, 메시지, 양식)
- 기존 UX 텍스트 개선 및 명확성/효과성 강화
- 오류 메시지, 알림 또는 성공 메시지 생성
- 대화 흐름 및 온보딩 경험 설계
- 제품의 목소리와 어조 정의
- 제품 콘텐츠의 일관성 및 유용성 검토

## **핵심 UX 글쓰기 원칙**

### **네 가지 품질 기준**

모든 UX 텍스트는 다음 조건을 충족해야 합니다.

1. **목적 지향적**
    
    — 사용자 또는 기업이 목표를 달성하도록 돕습니다.
    
2. **간결함**
    
    — 의미를 유지하면서 불필요한 단어를 줄입니다.
    
3. **대화체**
    
    — 로봇 같지 않고 자연스럽고 사람처럼 말합니다.
    
4. **명확함**
    
    — 모호함 없이 정확하고 쉽게 전달합니다.
    

### **핵심 모범 사례**

1. **간결함**
    - 한 줄에 최대 40~60자까지 사용하세요
    - 모든 단어에는 역할이 있어야 해요
    - 빽빽한 텍스트를 읽기 쉬운 단위로 나누세요
    - 중요 정보를 먼저 제시하세요
2. **명쾌함**
    - 누구나 쉽게 이해할 수 있는 수준의 언어를 사용하세요
    (일반 사용자: 중학생 수준, 전문가 대상: 고등학생 수준)
    - 어려운 전문 용어나 관용 표현은 피하세요
    - 같은 의미는 항상 같은 단어로 일관되게 표현하세요
    - 행동이 분명하게 드러나는 구체적인 동사를 사용하세요
3. **대화체**
    - 말하듯 자연스럽게 표현하세요
    - 능동형 중심으로 문장을 구성하세요
    - 문장을 자연스럽게 만드는 요소(조사와 어미 등)를 충분히 사용하세요
    - 딱딱한 표현 대신 사람처럼 부드럽게 말하세요
4. **사용자 중심**
    - 기능보다 사용자가 얻는 이점에 집중하세요
    - 사용자가 궁금해할 내용을 미리 예상해 답하세요
    - 2인칭 표현(“당신”)으로 직접적으로 말하세요
    - 사용자의 언어와 사고방식에 맞춰 표현하세요

## **UX 텍스트 패턴**

인터페이스 요소에 이러한 일반적인 패턴을 적용합니다.

### **1. 제목**

- **목적**: 사용자가 현재 위치를 바로 이해하도록 안내
- **형식**: 짧은 명사형으로 간결하게 작성
- **유형**: 브랜드명, 콘텐츠명, 카테고리명, 작업명
- **예시**: "계정 설정", "내 라이브러리", "새 게시물 작성"

### **2. 버튼 및 링크**

- **목적**: 사용자가 바로 행동할 수 있도록 유도
- **형식**: 능동형 동사 중심으로 간결하게 작성
- **패턴**: `[동사] + [대상]`
- **예시**: "변경 사항 저장", "계정 삭제", "세부 정보 보기"
- **피해야 할 문구**: “확인”, “제출”, “여기를 클릭하세요”처럼 의미가 모호한 표현

### **3. 오류 메시지**

- **목적**: 문제점을 설명하고 해결 방법까지 안내
- **형식**: 공감하면서도 명확하고, 바로 행동할 수 있게 작성
- **패턴**: `[무엇이 잘못되었는지] → [이유/상황] → [해결 방법]`
- **오류 메시지 유형**
    - **유효성 검사 오류(인라인)**
        - **목적**: 입력 중 발생한 오류를 즉시 알려주고 수정을 도움
        - **표시** **시점**: 입력 중이거나 입력을 마친 직후(포커스 해제 시) 표시
        - **형식**: 짧고 구체적으로, 어떻게 수정해야 하는지 안내
        - **패턴**: `[필드] + [구체적인 조건]`
        - **예시**
        : “이메일에 @를 포함해 주세요”
        : “비밀번호는 8자 이상 입력해 주세요”
        : “미래 날짜를 선택해 주세요”
        - **위치**: 입력 필드 아래 또는 옆에 표시합니다
    - **시스템 오류 (모달/배너)**
        - **목적**: 문제 상황을 설명하고 해결 방법을 안내
        - **표시 시점**: 서버 작업이 실패한 직후 표시
        - **형식**: 무엇이 실패했는지 → 이유 → 해결 방법 순으로 전달
        - **패턴**: `[실패 내용] → [원인] → [해결 방법]`
        - **예시**
        : “결제에 실패했습니다. 카드 결제가 거부되었습니다. 다른 결제 수단을 사용해 보세요”
        : “변경 사항을 저장할 수 없습니다. 연결이 끊어졌습니다. 다시 연결한 뒤 시도해 주세요”
        : “업로드에 실패했습니다. 파일 크기가 너무 큽니다. 10MB 미만 파일을 선택해 주세요”
        - **위치**
        : 모달 창 또는 눈에 잘 띄는 배너에 표시
    - **차단 오류(전체 화면)**
        - **목적**: 사용을 막는 문제를 명확히 알리고, 해결 방법을 안내
        - **표시 시점**: 앱 실행 시 또는 특정 기능에 접근할 때 표시
        - **형식**: 무엇이 차단되었는지 → 이유 → 필요한 행동 순으로 전달
        - **패턴**: `[차단된 내용] → [이유] → [필요한 조치]`
        - **예시**
        : “업데이트가 필요합니다. 현재 버전은 더 이상 지원되지 않습니다. 계속하려면 지금 업데이트하세요”
        : “구독이 만료되었습니다. 계정이 일시 중지되었습니다. 이용하려면 구독을 갱신하세요”
        : “인증이 필요합니다. 기능을 사용하려면 이메일을 인증하세요. 받은 편지함을 확인해 주세요”
        - **위치**: 전체 화면 또는 큰 모달로 표시합니다
    - **권한 오류**
        - **목적**: 권한이 필요한 이유를 설명하고 허용을 유도
        - **표시 시점**: 해당 기능을 처음 사용할 때 표시
        - **형식**: 사용자 혜택 → 필요한 권한 순으로 전달
        - **패턴**: `[사용자 혜택] → [필요한 권한]`
        - **예시**
        : “주문 상태를 알림으로 받아보세요. 알림을 허용해 주세요”
        : “근처 매장을 확인해 보세요. 위치 정보 접근을 허용해 주세요”
        : “사진을 안전하게 백업하세요. 저장 권한을 허용해 주세요”
        - **위치**: 해당 기능을 사용하는 화면 내에서 자연스럽게 표시
    - **피해야 할 것들**
        - 설명 없이 코드만 보여주는 메시지 (“오류 403”)
        - 의미가 모호한 오류 메시지 (“잘못된 입력”, “허용되지 않는 문자”)
        - 딱딱하고 기계적인 표현 (“오류가 발생했습니다”)
        - 해결 방법이 없는 메시지 (다음 행동을 안내하지 않음)
        - 원인이 불분명한 표현 (“문제가 발생했습니다”, “뭔가 잘못됐습니다”)

### **4. 성공 메시지**

- **목적**: 작업이 완료되었음을 명확하게 알려줌
- **형식**: 과거형으로, 구체적이고 긍정적인 어조로 전달
- **패턴**: `[수행한 작업] + [결과/이점]`
- **예시**: "변경 사항이 저장되었습니다", "이메일이 전송되었습니다", "프로필이 업데이트되었습니다"

### **5. 빈 상태**

- **목적**: 콘텐츠가 없을 때 다음 행동을 안내
- **유형**: 최초 사용, 사용자 삭제, 결과 없음
- **형식**: 상황 설명 + 행동 유도(CTA)
- **예시**: "아직 메시지가 없습니다. 팀과 소통하려면 대화를 시작하세요."

### **6. 폼 필드**

- **레이블**: 입력 내용을 한눈에 이해할 수 있게 짧은 명사형으로 작성 (예: "이메일 주소", "전화번호")
- **지시사항**: 동사로 시작하고, 왜 필요한지 함께 설명
- **플레이스홀더**: 예시 입력이 필요한 경우에만 제한적으로 사용 (예: “[name@example.com](mailto:name@example.com)”)
- **도움말 텍스트**: 중요도에 따라 항상 표시하거나, 필요할 때만 보여줌

### **7. 알림**

- **목적**: 적절한 시점에 중요한 정보를 전달
- **유형**: 사용자 주의를 요구하는 알림 / 비교적 부담이 적은 알림
- **형식**: 동사로 시작하는 제목 + 상황 설명
- **예시**: “업데이트가 필요합니다. 계속하려면 최신 버전을 설치하세요”

## **목소리와 어조**

### **1. 목소리 (일관된 브랜드 말투)**

보이스는 제품 전반에 일관되게 드러나는 브랜드의 말투와 성격입니다. 다음 기준으로 정의합니다:

- **핵심 개념**: 브랜드를 설명하는 3~5가지 핵심 가치나 원칙을 정리
- **보이스 특성**: 각 핵심 개념을 구체적으로 보여주는 형용사로 표현
- **Do / Don’t 예시**: 어떤 표현을 사용하고 피해야 하는지 실제 문장으로 정리

※ 보이스 차트 작성은 `references/voice_chart_template.md`를 참고하세요

### **2. 어조 (상황에 맞게 조정)**

어조는 상황에 따라 말투를 조절하는 방식입니다. 브랜드의 기본 말투는 유지하되, 사용자의 상황과 감정에 맞게 다르게 표현합니다.

- **어조 결정 요소**
    - **목적**: 사용자가 이 텍스트를 보는 이유에 맞게 표현 (정보 제공, 행동 유도, 확인 등)
    - **맥락**: 사용자가 현재 하고 있는 작업 흐름에 맞춰 표현 (학습, 작업 완료, 오류 해결 등)
    - **감정 상태**: 사용자의 감정을 고려해 공감하거나 명확하게 안내 (좌절, 기대, 혼란, 불안 등)
    - **중요도**: 행동의 영향도에 따라 어조의 무게를 조절 (가벼운 설정 vs 중요한 결정)
- **사용자 감정 상태에 따른 어조 조절**
    - **좌절감** (오류, 실패, 방해 상황)
        - 공감하고 해결에 집중합니다
        - 사용자를 탓하지 않고 문제를 인정합니다
        - 바로 실행할 수 있는 해결 방법을 제시합니다
        - 예시: “결제에 실패했습니다. 카드 결제가 거부되었습니다. 다른 결제 수단을 사용해 보세요”
    - **혼란스러움** (첫 사용, 복잡한 기능)
        - 차분하고 친절하게 설명합니다
        - 단계를 나누어 이해를 돕습니다
        - 맥락과 필요한 안내를 함께 제공합니다
        - 예시: “은행 계좌를 연결하면 지출 내역을 확인할 수 있어요. 차근차근 안내해 드릴게요”
    - **자신감 있음** (반복 사용, 익숙한 작업)
        - 간결하고 빠르게 전달합니다
        - 불필요한 설명은 줄입니다
        - 결과만 명확하게 보여줍니다
        - 예시: “저장되었습니다”
    - **신중함** (중요한 결정, 데이터 손실 가능)
        - 신중하고 명확하게 안내합니다
        - 결과와 영향을 분명히 전달합니다
        - 사용자의 선택을 존중합니다
        - 예시: “계정을 삭제하시겠습니까? 모든 데이터가 삭제되며 복구할 수 없습니다”
    - **성공적인** (작업 완료, 성과 달성)
        - 긍정적이고 가볍게 격려합니다
        - 과하지 않게 성취를 전달합니다
        - 간단한 축하 메시지를 포함합니다
        - 예시: “프로필이 업데이트되었습니다. 변경 사항이 적용되었습니다”
- **콘텐츠 유형별 어조 조정**
    - **오류 메시지**: 공감적이고, 안심시키며, 해결책에 초점을 맞춘 메시지
        - 사용자를 탓하지 않습니다
        - 어떤 문제가 발생했는지 명확히 설명합니다
        - 다음에 무엇을 해야 할지 분명하게 안내합니다
    - **성공 메시지**: 긍정적이고 구체적이며 격려적인 메시지
        - 어떤 작업이 완료됐는지 명확히 알려줍니다
        - 행동의 중요도에 맞는 수준으로 표현합니다
        - 간결하고 이해하기 쉽게 작성합니다
    - **지침**: 명확하고 직접적으로, 도움이 되게 전달
        - 행동부터 제시합니다
        - 필요할 경우 이유를 함께 설명합니다
        - 간단한 단계로 나누어 안내합니다
    - **온보딩**: 초대, 격려, 간결함
        - 부담 없이 환영하고 자연스럽게 유도합니다
        - 사용자에게 제공되는 가치를 중심으로 설명합니다
        - 초기 행동과 성과를 가볍게 격려합니다
    - **확인 사항**: 진지하고 투명하게, 사용자의 선택 존중
        - 결과와 영향을 명확히 설명합니다
        - 사용자를 유도하거나 압박하지 않습니다
        - 되돌릴 수 있는 선택지를 제공합니다
    - **빈 상태**: 긍정적인 톤으로 상황 설명하고, 다음에 할 행동 안내
        - 왜 비어 있는지 이유를 설명합니다
        - 다음에 할 행동을 분명하게 제시합니다
        - 사용자가 시작할 수 있도록 격려합니다

## **편집 과정**

UX 텍스트를 네 단계에 걸쳐 편집합니다:

### **1단계: 목적성 점검**

- 사용자의 목표 달성에 도움이 되는지 확인합니다
- 비즈니스 목표에 기여하는지 확인합니다
- 사용자에게 전달되는 가치가 명확한지 점검합니다
- 예상되는 우려를 미리 반영하고 해결합니다

### **2단계: 간결성 점검**

- 불필요한 단어를 제거합니다
- 중복된 내용은 하나로 정리합니다
- 모든 단어가 필요한지 다시 확인합니다
- 중요한 정보부터 먼저 전달합니다

### **3단계: 대화체 점검**

- 소리 내어 읽었을 때 자연스러운지 확인합니다
- 가능하면 능동형으로 표현합니다 (필요한 경우만 수동형 사용)
- 문장이 부드럽게 이어지도록 연결어를 사용합니다
- 내부 용어나 어려운 표현은 피합니다

### **4단계: 명확성 점검**

- 구체적이고 정확한 동사를 사용합니다
- 동일한 의미는 같은 용어로 일관되게 표현합니다
- 읽기 쉬운지 점검합니다 ([Hemingway Editor](https://hemingwayapp.com/), [Flesch-Kincaid](https://hemingwayapp.com/articles/readability/flesch-kincaid-readability-test))
- 모호한 표현 없이 명확하게 전달합니다

## **워크플로우**

1. **맥락 이해**
    - 사용자 목표와 요구를 파악합니다
    - 비즈니스 목표를 확인합니다
    - 기술적 제약을 고려합니다
    - 사용자의 감정 상태를 이해합니다
2. **초안 작성**
    - 실제로 말하듯 자연스럽게 시작합니다
    - 상황에 맞는 패턴을 적용합니다
    - 보이스와 어조를 반영합니다
    - 중요한 정보부터 먼저 전달합니다
3. **반복 개선**
    - 목적성 → 간결성 → 대화체 → 명확성 순으로 점검합니다
4. **테스트 및 검증**
    - 팀과 함께 검토합니다
    - 가능하면 사용자 테스트를 진행합니다
    - 이해도와 과제 완료율을 기준으로 개선합니다

## **UX 글쓰기에서의 접근성**

접근성 높은 콘텐츠는 모든 사용자가 쉽게 이해하고 사용할 수 있도록 돕습니다. 
보조 기술을 사용하는 사용자도 불편 없이 제품과 상호작용할 수 있게 합니다.

### **1. 접근성 핵심 원칙**

- **스크린 리더 최적화**
    - 모든 인터랙션 요소에 명확한 레이블을 제공합니다 (예: “제출” → “양식 제출”)
    - 링크는 목적이 드러나도록 구체적으로 작성합니다 (예: **X**-"여기를 클릭하세요" **O**-“가격 정보 보기”)
    - 오류 메시지는 필드와 함께 읽히도록 구조를 설계합니다
    - 시각 정보만으로 부족할 경우 aria-label을 활용합니다 (예: aria-label="닫기")
- **인지적 접근성**
    - 한 문장은 8~14단어 내로 작성해 이해하기 쉽게 만듭니다 (8단어 = 100% 이해, 14단어 = 90%)
    - 복잡한 내용은 읽기 쉬운 단위로 나눕니다
    - 명확한 제목과 구조로 정보를 정리합니다
    - 일관되고 예측 가능한 패턴으로 혼란을 줄입니다
- **다중 모드 커뮤니케이션**
    - 의미를 전달할 때 색상에만 의존하지 않습니다
    - 색상과 함께 텍스트나 아이콘을 함께 사용합니다 (예:"🚫 이메일을 입력하세요")
    - 아이콘과 이미지에는 대체 텍스트를 제공합니다 (예: aria-label="닫기")
    - 충분한 색상 대비를 유지합니다 (WCAG AA 최소 기준: 4.5:1).
- **모두를 위한 쉬운 언어**
    - 누구나 이해할 수 있는 쉬운 수준으로 작성합니다 (중학생 정도)
    - 처음 사용하는 용어는 간단히 설명합니다
    - 관용구나 은유, 문화적 표현은 피합니다
    - 일상에서 자주 쓰는 단어를 사용합니다

### **2. 접근성 패턴 예시**

- **버튼**
    - ❌ 나쁨: “제출” (맥락이 부족함)
    - ✅ 좋음: "신청서 제출"
- 링크
    - ❌ 나쁨: "자세한 정보를 보려면 여기를 클릭하세요"
    - ✅ 좋음: “개인정보 처리방침 보기”
- **오류 메시지**
    - ❌ 나쁨: 빨간 글씨로 “유효하지 않음”만 표시
    - ✅ 정상: “오류: 이메일에 @를 포함해 주세요” (아이콘 함께 사용)
- **폼 레이블**
    - ❌ 나쁨: 플레이스홀더만 있는 입력 필드
    - ✅ 좋음: 명확한 레이블 제공 + 필요 시 예시(플레이스홀더) 함께 사용

## **UX 텍스트 벤치마크**

연구 기반 지표를 활용해, 더 이해하기 쉽고 효과적인 UX 텍스트를 작성합니다.

### **1. 문장 길이 목표**

- **콘텐츠 유형별**
    - **버튼/CTA**: 2~4단어 (최대 6단어)
    - **제목**: 3~6단어 (최대 40자)
    - **오류 메시지**: 12~18단어 (해결 방법 포함)
    - **지침**: 14단어 권장 (최대 20단어)
    - **본문**: 문장당 평균 15~20단어
    - **알림**: 제목 + 본문 총 10~15단어
- **이해도 기준**
    - **8단어 이하:** 이해도 매우 높음 (약 100%)
    - **14단어 이하:** 이해도 높음 (약 90%)
    - **25단어 이하:** 권장 최대 길이 (이 이상은 이해도 저하)

### **2. 문자 및 줄 길이**

- **줄 길이**: 한 줄 40~60자
- **버튼 라벨**: 15~25자
- **페이지 제목**: 30~50자
- **알림 제목**: 35~45자 내로 핵심만 전달

### **3. 읽기 수준 지침**

- **대상에 따른 기준**
    - **일반 사용자**: 중학생 수준
    - **전문 도구**: 고등학생 초반 수준
    - **기술 제품**: 고등학생 수준
    - **전문 분야**: 필요 시 더 높은 수준 적용
- **가독성 점검 도구**
    - [**Hemingway Editor**](https://hemingwayapp.com/): 복잡한 문장 확인
    - [**Readable.com**](https://readable.com/): 가독성 점수 제공
    - [**Microsoft Word**](https://support.microsoft.com/en-au/office/get-your-document-s-readability-and-level-statistics-85b4969e-e80a-4777-8dd3-f7fc3c8b3fd2): Flesch-Kincaid 점수 확인

## 자주 하는 실수와 개선 방법

- 수동형 대신 능동형으로 명확하게 표현합니다
- “제출”, “확인”처럼 모호한 버튼 대신 의미를 드러냅니다
- 오류 메시지에서 사용자를 탓하지 않습니다
- 중요한 상황에서는 과한 유머를 사용하지 않습니다
- 같은 의미는 항상 같은 용어로 일관되게 사용합니다
- 필요한 안내는 숨기지 않고 분명하게 제공합니다
- 시스템 중심이 아닌 사용자 관점으로 표현합니다
- 불필요한 표현을 줄이고 간결하게 작성합니다
- 딱딱한 문장 대신 자연스럽고 사람처럼 말합니다
- 색상만으로 의미를 전달하지 않습니다
- “여기를 클릭하세요” 대신 목적이 드러나는 링크를 사용합니다
