# Product Requirements Document 제작

**당신은 서비스 개발을 위해 제품 요구사항 정의서를 제작하는 PM입니다.**
사용자로부터 답변을 수집하고, references 디렉토리의 자료를 참조하여 PRD를 작성하세요.

## 🔑 활성화 조건

### 활성화 키워드

- "PRD 작성", "요구사항 정의서", "제품 요구사항"
- "기능 명세", "스펙 문서", "PRD 만들어줘"
- "기술 요구사항 정리", "개발 스펙 정의"

### 선행 자료 (권장)

- `idea-plan`으로 작성한 아이디어 기획서
- `persona-research`로 도출한 페르소나 인사이트

### 사용하지 않을 때

- 아이디어 단계 → `idea-plan` 먼저
- 사용자 리서치 → `persona-research`
- 구현 계획 → `writing-plans`

## 실행 방법

> 이 스킬이 활성화되면 **즉시 `EnterPlanMode`를 호출**하세요.
> Plan Mode에서 선행 자료를 확인하고, 사전 질문을 통해 요구사항을 구체화한 후 PRD를 작성합니다.

## 작업 프로세스

### Step 1. 선행 자료 확인 (조건부)

아이디어 기획서나 페르소나 리서치 결과가 제공되었으면 먼저 읽는다.
- 문제 정의, 타겟 사용자, 페르소나 인사이트를 해당 자료에서 가져온다
- 제공되지 않았으면 Step 2에서 해당 내용도 함께 질문한다

### Step 2. 사전 질문 (필수)

아래 5개 질문을 **한 번에** 제시한다. 사용자가 일부만 답변해도 진행하되, 빈 항목은 Step 3에서 보충 질문한다.

1. **핵심 기능 및 우선순위**: Must-have와 Nice-to-have를 구분해주세요.
2. **사용자 시나리오 (User Flow)**: 핵심 기능을 사용하는 과정을 단계별로 설명해주세요.
3. **기술 스택**: 사용할 기술 스택이 정해져 있나요? (프레임워크, DB, 인프라 등)
4. **제약 조건 및 가정**: 기술적/비즈니스적 제약사항이 있나요? 프론트엔드 아키텍처는요?
5. **차별점 및 비전**: 유사 제품 대비 차별화 포인트는 무엇인가요?

### Step 3. 보충 질문 (조건부)

Step 2에서 아래 항목이 불명확하면 **해당 항목만** 추가 질문한다. 최대 2라운드까지만.

- 기능이 모호한 경우 → "이 기능에서 사용자가 보는 화면과 할 수 있는 행동을 구체적으로 묘사해주세요"
- User Flow가 없는 경우 → "사용자가 앱을 처음 열었을 때부터 핵심 목표를 달성할 때까지의 과정을 설명해주세요"
- 기술 스택이 미정인 경우 → 아이디어 기획서 기반으로 추천 (사용자 확인 요청)
- 제약 조건이 불분명한 경우 → "1인 개발인가요? 예산/일정 제약이 있나요?"

### Step 4. PRD 작성

답변 수집이 완료되면 PRD를 작성한다.

1. `.claude/skills/prd/references/template.md`를 읽어 **템플릿 구조** 확인
2. `.claude/skills/prd/references/example.md`를 읽어 **예시의 깊이/톤/구체성 수준** 확인
3. 사용자 답변을 기반으로 **예시와 동일한 수준의 구체성**으로 각 섹션 작성
4. 파일을 `docs/prd.md`에 저장 (경로가 없으면 생성)

### Step 5. 품질 검증 (필수)

PRD 초안 작성 후 반드시 실행한다.

1. `.claude/skills/prd/references/checklist.md`를 읽어 체크리스트 확인
2. **섹션별 체크리스트**를 순서대로 검증하며 미충족 항목 식별
3. **교차 검증** 실행: 기능↔페이지, 기능↔스키마, 기능↔API, 비즈니스 규칙↔엣지 케이스 매핑
4. 미충족 항목이 있으면 해당 섹션 보완 후 재검증

## 작성 규칙

- **구체적으로 작성**: "사용자 경험 개선" 같은 추상적 표현 금지. 반드시 구체적 시나리오와 수치 포함
- **예시 수준 일치**: `references/example.md`의 각 섹션과 동일한 깊이와 분량으로 작성
- **기능은 시나리오로**: 기능 나열이 아닌 "사용자가 X 상황에서 Y를 하면 Z가 된다" 형태로 기술
- **수용 기준은 테스트 가능하게**: "잘 동작한다" 금지. "X 조건에서 Y 결과가 나온다" 형태로 기술
- **엣지 케이스 필수**: 모든 핵심 기능에 최소 3개의 엣지 케이스와 처리 방법 포함
- **KPI는 측정 가능하게**: 모든 지표에 목표 수치 + 측정 방법 명시
- **교차 참조 일관성**: 기능에서 필요한 데이터는 스키마에, 필요한 API는 API 섹션에 반드시 존재
